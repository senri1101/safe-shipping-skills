# Deep Dive Index

Warning や Blocker が出た箇所だけ深掘りするための質問集。すべてを毎回使わない。

## A. アーキテクチャ設計

### A-1. アーキテクチャスタイル
- 今の構成は、想定規模に対して過剰か不足か
- モノリス、モジュラーモノリス、マイクロサービス、サーバーレスのどれで、なぜその形か
- UI、業務ロジック、外部 I/O の境界は明確か

### A-2. ドメインモデリング
- このプロダクトの中心概念は何か
- エンティティ、値オブジェクト、集約の境界を説明できるか
- どの不変条件を絶対に守るべきか

### A-3. モジュール分割・依存関係
- 循環依存はないか
- 依存方向は、変わりにくい層へ向かっているか
- 変更頻度の高い機能が、広範囲へ散弾銃的変更を起こしていないか

## B. データとインターフェース

### B-1. データモデリング
- 何を保存し、何を導出値にするか
- 1対多、多対多、履歴保持、削除方針は固まっているか
- 後から増えやすい属性を固定列に押し込んでいないか

### B-2. API設計
- 契約は何か。誰が何を送ると何が返るかを説明できるか
- バージョニング、ページネーション、エラー形式、冪等性キーは必要か
- over-fetching / under-fetching を起こしていないか

### B-3. 整合性・トランザクション
- 途中で失敗したら、何が部分成功として残るか
- 二重実行や再送が起きたとき、二重課金や二重登録にならないか
- トランザクション境界と Outbox / Saga / 冪等性の責任位置はどこか

### B-4. 非同期処理
- delivery semantics は何を期待しているか
- 重複受信時に安全か
- DLQ、可視性タイムアウト、リトライ回数は設計されているか

### B-5. キャッシュ
- キャッシュが古いと何が壊れるか
- 無効化、TTL、スタンピード対策、ホットキー対策はあるか
- CDN とアプリ内キャッシュの責任分担は明確か

## C. UX設計

### C-1. ユーザー理解
- 誰が何を達成したいのか
- 失敗時にもっとも困る利用者は誰か
- 仮説だけでなく、観察やヒアリングの根拠はあるか

### C-2. 情報設計
- 情報の並びは利用者のメンタルモデルに沿っているか
- 導線が深すぎないか、重要機能が埋もれていないか
- 検索、フィルタ、一覧、詳細の粒度は妥当か

### C-3. インタラクション設計
- 完了、失敗、再試行、キャンセル、空状態で迷わないか
- 取り消しや復旧が必要な操作に Undo / 再試行があるか
- 長時間処理やバックグラウンド処理の状態が可視化されているか

### C-4. ヒューリスティクス
- システム状態は見えるか
- エラーを防ぐ設計になっているか
- 利用者の記憶負荷を増やしていないか

### C-5. アクセシビリティ
- キーボードだけで使えるか
- スクリーンリーダーで主要操作を追えるか
- 色覚差や認知負荷への配慮があるか

### C-6. UXメトリクス
- 成功と失敗を何で測るか
- 離脱、再試行、完了率、時間などの指標はあるか
- ログやイベントで計測可能か

## D. 非機能・運用

### D-1. スケーラビリティ
- 何人、何件、何 QPS までを想定しているか
- ボトルネックは DB、外部 API、ネットワーク、キューのどこか
- ステートフルな要素が水平拡張を妨げていないか

### D-2. 可用性・レジリエンス
- 単一障害点はどこか
- タイムアウト、リトライ、サーキットブレーカー、バックオフはあるか
- 依存先が落ちたら、どう degrade するか

### D-3. 可観測性
- 障害時に `誰が / 何を / いつ / どこで` を追えるか
- ログ、メトリクス、トレースの最低限はあるか
- アラートが多すぎて埋もれないか

### D-4. 環境・インフラ設計
- 開発、ステージング、本番は分かれているか
- IaC や設定コード化で再現可能か
- 「本番でちょっと試す」運用が入り込んでいないか

### D-5. データマイグレーション
- カラム追加、削除、型変更の順序を説明できるか
- expand-migrate-contract か、メンテナンスウィンドウか
- 件数比較、チェックサム、ロールバック方法があるか

## E. セキュリティ設計

### E-1. セキュリティ原則
- 最小権限、デフォルト拒否、監査証跡は守られているか
- 「侵害前提」で考えたとき、止血点はどこか
- 守るべき資産と攻撃面は何か

### E-2. データ保護・マルチテナント
- どのデータが機密、内部、公開か
- テナント境界はどこで切り、どこで検証するか
- 共有基盤上で漏えいしない理由を説明できるか

### E-3. LLMセキュリティ
- LLM への入力はどこを通り、何が保持されるか
- 出力を無検証で実行していないか
- プロンプトインジェクション、ツール汚染、過剰権限の対策はあるか

## F. UI設計・フロントエンド

### F-1. ビジュアルデザイン
- 視覚的優先順位は明確か
- タイポグラフィ、コントラスト、余白に一貫性があるか
- 情報密度は利用シーンに合っているか

### F-2. デザインシステム
- トークン、コンポーネント、命名規則は揃っているか
- 既存プロダクトの表現を壊していないか
- 1 回限りの画面でも再利用可能な粒度が保たれているか

### F-3. コンポーネント設計
- 表示責務と状態責務が混線していないか
- props / children / slot の契約はわかりやすいか
- 再利用のために設定過剰になっていないか

### F-4. スタイリング
- スタイリング方針は 1 つに寄っているか
- ブレークポイント、Container Queries、ダークモードの責任位置はどこか
- 変数化すべき値が散っていないか

### F-5. 状態管理
- ローカル状態、URL 状態、サーバー状態、フォーム状態を分離できているか
- 楽観更新や再取得時の整合性は保てるか
- 状態の二重保持で矛盾が起きていないか

### F-6. フロントエンド性能
- LCP、INP、CLS を悪化させる要素は何か
- 画像、JS、フォント、SSR/CSR 戦略は妥当か
- 体感速度と実測値の両方を見ているか

### F-7. 実装アクセシビリティ
- セマンティック HTML を使っているか
- フォーカス遷移、ラベル、aria-live を確認したか
- キーボード操作だけで主要フローを完走できるか

### F-8. アニメーション・インタラクション
- 動きは意味を持っているか
- `transform / opacity` で済む範囲を超えていないか
- `prefers-reduced-motion` に対応しているか

## G. 設計原則とパターン

### G-1. 設計原則
- この責務はその型・そのモジュールに置くべきか
- DRY、KISS、YAGNI、関心の分離を破っていないか
- 条件分岐や例外処理が増殖していないか

### G-2. GoF パターン
- パターンを使う理由があるか
- 手続きで十分なのに抽象化しすぎていないか
- Proxy、Strategy、Decorator などが複雑さの温床になっていないか

### G-3. エンタープライズパターン
- Repository、Service、Unit of Work の境界は適切か
- Transaction Script と Domain Model のどちらが妥当か
- Specification や Facade で複雑性を下げられるか

## H. コード品質

### H-1. クリーンコード
- 命名だけで意図が読めるか
- 関数は 1 つのことをしているか
- コメントは「なぜ」を補足しているか

### H-2. リファクタリング
- 巨大クラス、長すぎる関数、データの群れはないか
- 条件分岐の増殖を型やポリモーフィズムで閉じられないか
- 今直すべき臭いと、後回しにしてよい臭いを分けているか

### H-3. 型設計・契約
- 不正状態を型で表現不能にできるか
- nullable、option、result の扱いは妥当か
- 事前条件、事後条件、不変条件はどこで守るか

### H-4. 関数型の考え方
- 副作用は境界に寄せられているか
- イミュータブルに扱えるデータを破壊していないか
- 変換と I/O が混ざっていないか

### H-5. エラーハンドリング
- 回復可能なエラーと回復不能なエラーを分けているか
- リトライ対象を間違えていないか
- 例外を握りつぶしていないか

## I. テスト

### I-1. テスト戦略
- どこを自動化し、どこを人が見るか
- テストピラミッドやトロフィーのバランスは妥当か
- カバレッジだけを目的化していないか

### I-2. 機能テスト
- ハッピーパスだけでなく、失敗系、境界系、認可系を見ているか
- ユニット、統合、E2E の責務が分かれているか
- リグレッションの重要フローが固定されているか

### I-3. UI・ビジュアルテスト
- 画面崩れ、アクセシビリティ、レスポンシブを確認できるか
- 視覚差分は誤検知に埋もれていないか
- スクリーンショットだけで意味のない確認になっていないか

### I-4. 契約・境界テスト
- API 契約、OpenAPI、JSON Schema を検証しているか
- Consumer と Provider のズレを早期に見つけられるか
- エラー応答や権限エラーも契約に入っているか

### I-5. 非機能テスト
- 負荷、ストレス、耐久、セキュリティのどこまで必要か
- 小規模プロジェクトでも最低限の性能とセキュリティ確認をしているか
- リリース直前だけでなく設計段階から観点があるか

### I-6. テストダブル
- Mock を使いすぎて実装詳細を固定していないか
- Fake / Stub / Spy / Mock を区別しているか
- 本当に外部境界だけを差し替えているか

### I-7. テスト設計技法
- 同値分割、境界値分析、状態遷移を使えているか
- 認可や課金など、組み合わせ爆発をどう抑えるか
- 連打、再送、同時実行をケース化しているか

## J. 実装技術

### J-1. 並行処理・マルチスレッド
- 再入、連打、並列実行で race condition が起きないか
- lock、queue、async/await の境界は安全か
- TOCTOU や重複実行が起きる場所はどこか

### J-2. パフォーマンス
- N+1、フルスキャン、無制限ループ、巨大レスポンスはないか
- インデックス、EXPLAIN、圧縮、ページネーションを見ているか
- 早すぎる最適化と、遅すぎる無視を区別しているか

## K. プロセスとドキュメント

### K-1. ドキュメンテーション
- README、ADR、API ドキュメント、運用手順はあるか
- 変更理由が将来の保守者に伝わるか
- 図や文章がコードと矛盾していないか

### K-2. バージョン管理・開発プロセス
- コミット粒度は小さいか
- PR でレビュー可能な単位になっているか
- CI、ブランチ戦略、フィーチャーフラグは必要十分か

## L. セキュアコーディング

### L-1. 脅威モデリング
- 攻撃者、資産、入口、信頼境界を説明できるか
- STRIDE のどれが現実的な脅威か
- どの攻撃を「今回は扱わない」と判断したか

### L-2. 通信の保護
- TLS、HSTS、CSP、CORS、クリックジャッキング対策は妥当か
- `*` と `credentials` の危険な組み合わせがないか
- 社内向けでも通信路を過信していないか

### L-3. 暗号化・機密情報
- パスワードや秘密は適切な方式で保護されているか
- 鍵管理、ローテーション、KMS/Vault の責任分担は明確か
- ログや分析基盤へ平文機密が流れていないか

### L-4. 認証・セッション
- セッション、JWT、Cookie 属性、ログアウト失効は妥当か
- MFA やレート制限が必要な場面で省略していないか
- state / nonce / redirect_uri 検証が必要な認証フローを理解しているか

### L-5. 認可・アクセス制御
- 認証後に本人所有確認を毎回サーバー側でしているか
- テナント境界はどこで検証するか
- 403 と 404 の使い分けは情報漏えいを増やさないか

### L-6. 入力検証
- ホワイトリスト方式で検証しているか
- JSON Schema や OpenAPI で境界を定義できるか
- ファイルアップロードの種別、サイズ、保存先を安全に制御しているか

### L-7. インジェクション攻撃
- SQL、OS コマンド、テンプレート、LDAP、NoSQL の危険経路はどこか
- Raw query や文字列連結を使っていないか
- デシリアライズや外部テンプレート評価を無防備に行っていないか

### L-8. 出力エンコーディング
- HTML、属性、URL、JavaScript、CSS の文脈別エスケープをしているか
- `dangerouslySetInnerHTML` 相当を安全化しているか
- オープンリダイレクトや `javascript:` スキームを許していないか

### L-9. エラー処理
- 本番では内部詳細を返していないか
- 機密情報をログや監視へ送っていないか
- 監査・調査に必要な粒度だけを残しているか

### L-10. SSRF
- URL 入力を受ける機能はあるか
- メタデータサービス、リンクローカル、内部 IP への到達を防げるか
- DNS rebinding やリダイレクトをどう扱うか

### L-11. 依存関係
- lock file、脆弱性スキャン、EOL パッケージ棚卸しはあるか
- AI が提案したパッケージの実在と保守状況を確認したか
- ライセンス互換性を見たか

## M. 参考書籍・フレームワーク

### M-1. 設計・実装
- 今回の判断を支える標準や書籍は何か
- Clean Architecture、DDD、PoEAA など、借りた原則を誤用していないか
- チーム内で参照する基準文書を共有しているか

### M-2. セキュリティ
- OWASP Top 10、ASVS、Testing Guide、IPA、CERT、CWE のどれを参照すべきか
- 今回の脅威レベルに対して、どこまでを最低ラインにするか
- セキュリティを「後でまとめてやる」前提にしていないか
