# Stage Gates

Stage Gate は、現在段階に応じて追加で適用する。Hard Gate を先に終えてから使う。

## 0. Classification

目的:
- 「使い捨てか継続運用か」
- 「誰にどこまで公開するか」
- 「どのデータを扱い、事故時に誰が困るか」

確認項目:
- これは週末用の試作か、継続運用前提か
- 一般公開、限定公開、社内、個人用のどれか
- 規制対象や個人情報を含むか
- AI/LLM が関わるなら、出力を無検証で実行する経路があるか

この段階で `Blocker` にしやすいもの:
- 公開範囲と利用者が曖昧
- 扱うデータの機微度が説明できない
- 継続運用なのに「使い捨て前提」で設計しようとしている

## 1. Pre-build

目的:
- 作る前に、法務・コスト・公開条件・社内ルールを固める

確認項目:
- 料金体系を理解し、主要コスト源の試算があるか
- 予算アラート、使用上限、濫用防止策があるか
- 利用規約、個人情報、ライセンス、規制業法の確認があるか
- 社内データを使う場合、承認済み AI ツールとアカウントか
- 公開前提に応じた最小のセキュリティ要求が定義されているか

強い `Warning` になりやすいもの:
- コスト試算はあるが、リリース後の確認頻度が決まっていない
- プライバシーポリシーや利用規約確認が未文書化

## 2. Design

目的:
- 後から壊しにくい部分を先に固める

確認項目:
- エンティティ、関係性、削除方針、変わりやすい属性が整理されているか
- migration 順序、expand-migrate-contract、中間互換の考え方があるか
- トランザクション境界、冪等性、再試行、二重実行対策があるか
- 途中失敗時の一貫性と回復方法が定義されているか
- 性能前提があるか
  - 想定ユーザー数
  - 主要 read / write 量
  - 高コストクエリや N+1 になりそうな箇所
- 開発 / 本番 / ステージングの分離が設計時点で考慮されているか

この段階で `Blocker` にしやすいもの:
- データ所有関係が曖昧
- 認可境界と所有者確認が設計に存在しない
- 二重課金、二重登録、部分成功の失敗モードが未整理

## 3. Implementation

目的:
- 実装が設計意図を破っていないかを確認する

確認項目:
- 秘密情報がコード、履歴、ログ、プロンプトに出ていないか
- 依存関係が実在し、保守され、lock file で固定されているか
- サーバー側入力検証、認可、出力エンコード、エラーマスキングがあるか
- テスト戦略があるか
  - 自動テスト範囲
  - 人的確認範囲
  - 失敗系、境界系、認可系
- テスト名と仕様の対応が読めるか
- Git、PR、CI、レビュー、最低限のドキュメントがあるか
- 保守性が保たれているか
  - 同じロジックの重複
  - 巨大関数
  - 説明できないコード

この段階で `Blocker` にしやすいもの:
- 認可、秘密管理、入力検証、依存関係固定が欠ける
- 本番向けなのにテストもレビューもなく直接投入する

## 4. Release & Ops

目的:
- 出したあとに壊れたとき、見つけて止めて戻せる状態にする

確認項目:
- ログ、メトリクス、アラート、必要ならトレースがあるか
- 予算アラート、請求確認、濫用検知があるか
- バックアップ、復元、ロールバック、メンテナンス手順があるか
- リリース方式、切り戻し基準、責任者が明確か
- インシデント対応の初動が決まっているか
  - 止める
  - 記録する
  - 影響範囲を知らせる
  - 原因を掘る
  - 再発防止を仕組みに落とす
- 運用中なら、週次・月次のログ確認、コスト確認、権限棚卸しがあるか

この段階で `Blocker` にしやすいもの:
- 監視もアラートもなく、障害や侵害に気づけない
- 予算逸脱、データ破壊、migration 失敗時に止血できない
- リストア不能

## N/A の付け方

- 静的サイトで DB がなければ、migration / transaction は `N/A`
- 認証がない静的サイトなら Authz は `N/A` になり得る
- AI を使わないなら AI tool policy や prompt injection は `N/A`
- ただし `N/A` は「不要である理由」を 1 行で説明できるときだけ使う
