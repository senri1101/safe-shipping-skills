# Hard Gates

Hard Gate は、現在段階に関係なく最初に判定する。ここで `Blocker` が出たら、後続レビューより先に是正方針を明示する。

## H1. Security Baseline

- 何を見るか
  - 公開面、管理面、管理者操作、クラウド公開設定、通信保護、最小権限
- `Pass` の証拠
  - 公開境界が説明できる
  - デフォルト拒否または最小権限で構成されている
  - 本番公開設定がコードか管理画面で確認できる
- `Blocker`
  - 誰がアクセスできるか説明できない
  - 公開バケット、広すぎるネットワーク開放、万能権限がある
  - IaaS を使うのに境界設定と責任範囲が未確認
- `Warning`
  - 設定はありそうだが、レビュー、テスト、文書化が足りない
- 次に取るべき最小アクション
  - 公開面と管理面を列挙し、権限と露出設定を 1 つずつ確認する

## H2. Secrets Management

- 何を見るか
  - API キー、秘密鍵、トークン、env、CI secrets、再発行手順
- `Pass` の証拠
  - 秘密情報がコード・プロンプト・ログ・画像に直書きされていない
  - 環境ごとに鍵が分離されている
  - 失効・再発行手順がある
- `Blocker`
  - repo、履歴、画面キャプチャ、AI への質問文に秘密が残っている
  - 公開鍵と秘密鍵を混同している
  - 本番と検証で鍵を共用している
- `Warning`
  - 秘密は分離されているが、棚卸しやローテーション方針がない
- 次に取るべき最小アクション
  - 露出している秘密を無効化し、保管場所と利用箇所を一覧化する

## H3. Authorization And Tenant Isolation

- 何を見るか
  - ログイン後の所有者確認、RBAC/ABAC、IDOR、テナント境界
- `Pass` の証拠
  - サーバー側で毎回「このデータは本人または許可された主体のものか」を検証している
  - テナント境界の条件が明示されている
- `Blocker`
  - ログインだけで認可完了とみなしている
  - ID を差し替えると他人のデータに到達し得る
  - テナント境界がアプリ任せで、サーバー側検証がない
- `Warning`
  - 認可ロジックはあるが、自動テストやレビュー証跡が弱い
- 次に取るべき最小アクション
  - 所有者確認と権限確認を全 read / write パスで棚卸しする

## H4. Input Validation And File Handling

- 何を見るか
  - サーバー側バリデーション、スキーマ検証、ファイルアップロード、サイズ制限、パストラバーサル
- `Pass` の証拠
  - クライアントではなくサーバー側でも同じ制約を検証している
  - ファイル種別、サイズ、保存先、SVG などの危険形式の扱いが定義されている
- `Blocker`
  - クライアント側だけで入力制御している
  - 拡張子だけでファイル種別判定している
  - パスやファイル名を無加工で保存先に使っている
- `Warning`
  - バリデーションはあるが、エラー時の扱いや境界ケースが不足している
- 次に取るべき最小アクション
  - サーバー側スキーマとファイル制約を 1 箇所に集約する

## H5. Error Leakage And Safe Failures

- 何を見るか
  - 本番エラーメッセージ、スタックトレース、内部パス、SQL、機密情報露出
- `Pass` の証拠
  - ユーザー向けエラーは一般化され、詳細は内部ログにのみ残る
  - 本番と開発で表示内容が切り替わる
- `Blocker`
  - 本番利用者にスタックトレース、SQL、内部パス、ライブラリ詳細を返す
  - エラーログに秘密や個人情報をそのまま出す
- `Warning`
  - 一部は隠れているが、例外経路によって漏えいの可能性が残る
- 次に取るべき最小アクション
  - 本番エラー応答とログマスク方針を分ける

## H6. Logging And Alerting

- 何を見るか
  - アクセスログ、エラーログ、保存先分離、アラート、定期確認、監査証跡
- `Pass` の証拠
  - アクセスとエラーが記録される
  - アプリ本体とは別の保存先がある
  - 異常時の通知条件が定義されている
- `Blocker`
  - ログがない、または本番障害時に後追いできない
  - 侵害時に同時消失する単一保存先しかない
  - 機密情報をそのまま外部ログ SaaS に送る
- `Warning`
  - ログはあるが、見ていない、保持期間が不明、通知がない
- 次に取るべき最小アクション
  - 重要イベントのログ項目とアラート条件を最小セットで定義する

## H7. Cost Guardrail

- 何を見るか
  - 料金体系理解、従量課金、試算、予算アラート、ハードリミット
- `Pass` の証拠
  - 主要コスト源が列挙されている
  - 想定利用量で月額試算がある
  - 予算アラートまたは強制停止手段がある
- `Blocker`
  - 従量課金サービスを使うのに価格と上限を把握していない
  - LLM や画像生成が無制限に叩ける
  - 課金異常に気づく方法がない
- `Warning`
  - 初期試算はあるが、運用後の週次・月次確認が未整備
- 次に取るべき最小アクション
  - コスト源ごとに単価、無料枠、月額試算、アラート有無を 1 枚にまとめる

## H8. Legal, License, Privacy, AI Tool Policy

- 何を見るか
  - 利用規約、個人情報保護、規制業法、OSS ライセンス、生成物の権利、社内 AI 利用ルール
- `Pass` の証拠
  - 取得データと根拠法、利用規約、ライセンス制約、AI ツール利用条件を説明できる
  - 承認済みアカウントとツールを使っている
- `Blocker`
  - 規約違反のスクレイピングや無断データ利用がある
  - 個人情報や機密コードを未承認 AI に送る
  - GPL 等の制約を無視して商用利用する
- `Warning`
  - 大きな違反は見えないが、利用規約確認やプライバシー記載が未完了
- 次に取るべき最小アクション
  - 扱うデータ、根拠法、利用規約、ライセンス、AI ツール承認状況を一覧化する

## H9. Environment Separation

- 何を見るか
  - 開発、ステージング、本番の分離。DB、鍵、設定、データの混線
- `Pass` の証拠
  - 少なくとも開発と本番が分離されている
  - 環境ごとに DB、鍵、設定が分かれている
- `Blocker`
  - 本番で試す前提になっている
  - 開発環境から本番データや本番鍵をそのまま使っている
  - 誤操作で本番データを壊し得る
- `Warning`
  - 分離はあるが、ステージング未整備、権限混線、切替手順が曖昧
- 次に取るべき最小アクション
  - 環境一覧、接続先、鍵、デプロイ対象を可視化する

## H10. Backup, Restore, Rollback

- 何を見るか
  - 本番バックアップ、自動取得、復元手順、ロールバック、メンテナンス手順
- `Pass` の証拠
  - バックアップが自動化されている
  - 復元手順またはリハーサル記録がある
  - 変更失敗時のロールバック方法が定義されている
- `Blocker`
  - 本番データを失っても戻せない
  - migration やデータ更新にロールバック案がない
  - 直接本番コンソールで修正する運用になっている
- `Warning`
  - バックアップはあるが、復元確認やロールバック訓練がない
- 次に取るべき最小アクション
  - バックアップ取得、復元確認、ロールバック手順を 1 回通す
